# Exercise: Insurance FWA Assessor Portal

> **Time:** 60 minutes | **Level:** Intermediate Vibe Coding

---

> **Your mission:** Build a working AI-powered Fraud, Waste & Abuse (FWA) investigation portal — without writing a single line of code. You are the product owner. Claude Code is your developer.

---

## Before You Begin

**What you'll build:**
A web application where an insurance assessor can upload a claim document, run automated FWA checks powered by AI agents, review a risk dashboard, and chat with an AI about the findings.

**What you need:**
- Claude Code installed and running
- A browser (Chrome, Safari, or Edge)

**How this works:**
You will copy and paste prompts into Claude Code. Claude Code will build the app. You review it, ask for changes, and iterate — just like a PM briefing a developer.

---

| Step | What You'll Do | Time |

## Step 1: Project Setup & Claim Upload Screen (10 min)

### What you're building
A clean landing page where an assessor can upload a claim document (PDF or image) and enter basic claim details before kicking off the FWA review.

### Your prompt — copy and paste this into Claude Code:

```text
I want to build an Insurance FWA (Fraud, Waste & Abuse) Assessor Portal
as a self-contained web application (single HTML file with CSS and JavaScript).
I have no coding experience — you will handle all technical work.

This is a prototype — it will use simulated AI responses, not a real API.
No API keys are needed.

STEP 1 — Build the Claim Upload Screen:

DESIGN:
- Professional insurance look: deep navy #1B2A4A, white, and amber #F5A623
- Clean header with title "FWA Assessor Portal" and subtitle
  "AI-Powered Fraud, Waste & Abuse Investigation"
- Sans-serif font (Inter or Segoe UI)

UPLOAD SCREEN LAYOUT:
- A prominent file upload box (drag-and-drop style) that accepts
  PDF and image files (JPG, PNG). Label it "Upload Claim Document"
- Below the upload box, 4 input fields in a 2x2 grid:
    - Claim ID (text field)
    - Claimant Name (text field)
    - Claim Type dropdown: Motor, Home, Health, Commercial Property, Life
    - Claim Amount (EUR) (number field)
- A large button at the bottom labeled "Run FWA Investigation"
  in amber color
- When the file is uploaded, show the filename and a green checkmark
- The button should be disabled until a file is uploaded and all
  fields are filled in

For now, clicking "Run FWA Investigation" should show a loading spinner
with the text "Agents investigating...". We will wire up the actual
agents in the next step.

Make this a single HTML file I can open in my browser.
```

### Check before moving on
- [ ] Page opens in your browser
- [ ] You can drag and drop (or click to select) a file
- [ ] All 4 input fields appear
- [ ] The amber button appears and is disabled until fields are filled
- [ ] Clicking the button shows a loading spinner

## Step 2: Wire Up the Three AI Agents (15 min)

### What you're building
Three simulated AI agents that run in sequence when the assessor clicks "Run FWA Investigation":
1. **Claim Validity Agent** — checks if the claim is justified and consistent with the policy type
2. **Risk Scoring Agent** — scores the claim from 0–100 for fraud risk
3. **Action Recommendation Agent** — recommends what the assessor should do next

### Your prompt — copy and paste this:

```text
Now wire up the AI agent pipeline. When the user clicks
"Run FWA Investigation", the app should SIMULATE three agents
running in sequence. No real API calls — generate realistic
mock responses based on the claim details the user entered.

1. Read the uploaded filename and the claim form fields
2. Run THREE simulated agents in sequence, each with a
   short delay (1-2 seconds) to feel realistic

AGENT 1 — Claim Validity Agent
Generate a realistic JSON response based on the claim type and amount:
- validity_verdict: "Justified" | "Questionable" | "Unjustified"
  (vary based on claim amount — higher amounts more likely questionable)
- confidence: a percentage (0-100)
- key_findings: array of 3-5 bullet points (strings) relevant to the
  claim type
- red_flags: array of concerns (can be empty for small claims,
  1-3 items for larger ones)

AGENT 2 — Risk Scoring Agent
Generate a JSON response that considers Agent 1's findings:
- risk_score: number 0-100 (correlate with claim amount and
  Agent 1 findings)
- risk_level: "Low" | "Medium" | "High" | "Critical"
- risk_factors: array of contributing factors (strings)
- anomalies: array of specific anomalies detected (can be empty)

AGENT 3 — Action Recommendation Agent
Generate a JSON response based on all previous findings:
- recommended_action: one of "Approve", "Request More Information",
  "Refer to Special Investigations Unit", "Deny Claim"
- action_rationale: 2-3 sentence explanation (string)
- next_steps: array of 3-5 concrete steps for the assessor (strings)
- priority: "Routine" | "Urgent" | "Immediate"

Make the simulated responses internally consistent — if Agent 1 finds
red flags, Agent 2 should give a higher risk score, and Agent 3 should
recommend investigation.

LOADING EXPERIENCE:
- Show a step-by-step progress indicator as each agent runs:
  "Agent 1: Checking claim validity..."
  "Agent 2: Calculating risk score..."
  "Agent 3: Generating recommendations..."
- Each step should visually tick off when complete

Keep all results in a JavaScript variable called investigationResults
so we can display them in the next step.
```

### Check before moving on
- [ ] Upload a sample document (any PDF or image will do for testing)
- [ ] Fill in the claim fields
- [ ] Click the button — you should see the 3-step progress indicator
- [ ] Each agent ticks off as it completes
- [ ] Results are stored (you can check via browser console if curious)

## Step 3: Build the Results Dashboard (15 min)

### What you're building
After the agents finish, the app transitions to a results screen showing a clear, visual summary of all three agents' findings.

### Your prompt — copy and paste this:

```text
Now build the Results Dashboard that appears after all three agents
have finished. Replace the loading screen with this layout:

HEADER SECTION:
- Show "Investigation Complete" with a timestamp
- Show the Claim ID and Claimant Name from the inputs
- A colored banner based on the recommended action:
    - "Approve" → green banner
    - "Request More Information" → amber banner
    - "Refer to Special Investigations Unit" → orange banner
    - "Deny Claim" → red banner
- The recommended action and priority should be prominently displayed
  in this banner

THREE AGENT RESULT CARDS (displayed side by side in a row):

Card 1 — Medical Necessity (left):
- Title: "Medical Necessity Check"
- Large verdict badge (green/amber/red based on verdict)
- Confidence percentage shown as a progress bar
- List of key findings as bullet points
- Red flags section (show in red if any exist, hide if empty)

Card 2 — Risk Score (center):
- Title: "FWA Risk Score"
- A large circular score display showing the number (0-100)
- Color of circle: green <30, amber 30-60, orange 60-80, red >80
- Risk level label below the circle
- List of risk factors as bullet points
- Anomalies section (show in red if any, hide if empty)

Card 3 — Recommended Actions (right):
- Title: "Assessor Actions"
- Recommended action in a prominent badge
- Action rationale in a text box
- Numbered list of next steps for the assessor

BOTTOM SECTION:
- A "Download Report" button that exports the findings as a
  formatted text file
- A "New Investigation" button that resets the form
- Leave a placeholder div with id="chat-section" at the bottom
  with the text "Chat Assistant — Coming Soon"
  (we will build this in the next step)
```

### Check before moving on
- [ ] Results dashboard appears after agents finish
- [ ] All 3 agent cards are visible with the correct data
- [ ] The top banner color matches the recommendation

- [ ] Download Report button works

## Step 4: Add the Chat Assistant (10 min)

### What you're building
A chat panel at the bottom of the results page where the assessor can ask follow-up questions about the investigation findings.

### Your prompt — copy and paste this:

```text
Now build the Chat Assistant section. Replace the placeholder
div with id="chat-section" with a fully working chat interface.

CHAT UI:
- Section title: "Ask the FWA Assistant"
- Subtitle: "Ask follow-up questions about this investigation"
- A scrollable chat message window (300px tall) with:
    - An opening message from the assistant:
      "I've reviewed this claim. I found [risk level] risk with a score
       of [X]/100. The recommendation is to [action]. What would you
       like to know more about?"
      (populate this dynamically from the investigation results)
- A text input field at the bottom with placeholder
  "Ask a question about this claim..."
- A "Send" button in amber

CHAT BEHAVIOR:
- When the assessor sends a message, generate a simulated assistant
  response based on the investigation findings already in memory.
  The response should be relevant to the question asked and reference
  specific data from the investigation results.
- Show a typing indicator ("Assistant is thinking...") with a
  1-2 second delay to feel realistic
- Display the assistant's response in the chat window
- Keep conversation history so responses stay consistent
- Style user messages aligned right (navy background, white text)
  and assistant messages aligned left (light gray background)

EXAMPLE QUESTIONS the assessor might ask:
- "Why was this flagged as high risk?"
- "What documents should I request?"
- "Has this type of anomaly been seen before?"
- "Can you summarize the key concerns in one paragraph?"
```

### Final checks
- [ ] Chat panel appears at the bottom of results
- [ ] Opening message is pre-populated with the investigation findings
- [ ] You can type a question and get an AI response
- [ ] Conversation history is maintained across messages
- [ ] Messages are visually distinct (user vs assistant)

### Bonus 1 — Assessor Sign-Off & Audit Trail (~10 min)

*A real investigation tool needs accountability. Right now, findings just sit on screen. Make them official.*

```text
Add an assessor sign-off workflow to the results page:

SIGN-OFF PANEL (appears below the 3 agent cards):
- A text area labeled "Assessor Notes" where the user can type
  their own observations before finalizing
- A dropdown: "Final Decision" with options:
    Approve | Request More Information |
    Refer to SIU | Deny Claim | Override AI Recommendation
- If "Override AI Recommendation" is selected, show a mandatory
  text field: "Reason for override (required)"
- An "Assessor Name" text field
- A "Sign Off & Finalize" button in navy blue

BEHAVIOR:
- When signed off, lock the entire results page (make all fields
  read-only and gray out the buttons)
- Replace the sign-off panel with a green confirmation banner:
  "Investigation finalized by [Name] at [timestamp]"
- Update the Download Report to include the assessor's name,
  decision, notes, and timestamp at the top of the exported file
- Show a visual "FINALIZED" watermark stamp diagonally across
  the results cards
```

### Bonus 2 — Investigation Case File & Session History (~15 min)

*A portal without memory isn't a portal — it's a calculator. Give it persistence within the session.*

```text
Add a persistent Case File sidebar to the left of the main content
(collapsible on small screens):

SIDEBAR — "Case File":

    - Total cases reviewed
    - High/Critical risk count
    - Cases referred to SIU
    - Cases pending sign-off

SESSION SUMMARY BUTTON:
- At the bottom of the sidebar, add an "Export Session Summary"
  button that downloads a single CSV file containing one row per
  investigation with all key fields:
  Case Ref, Claimant, Type, Amount, Risk Score, Risk Level,
  AI Recommendation, Assessor Decision, Override Y/N, Assessor Name,
  Timestamp
```

### Bonus 3 — Comparator & Benchmark Mode (~15 min)

*The hardest challenge. Turn the portal from a single-case tool into an analytical platform.*

```text
Add a "Compare Cases" mode accessible from the sidebar.

COMPARE MODE:
- Allow the assessor to select exactly 2 cases from the session
  history (using checkboxes in the sidebar)
- A "Compare Selected" button appears when exactly 2 are checked
- Clicking it opens a full side-by-side comparison view:

LEFT COLUMN — Case A | RIGHT COLUMN — Case B
- Show both cases' key metrics aligned row by row:
    - Risk Score (with visual bar)
    - Medical Necessity Verdict
    - Number of red flags
    - Number of anomalies
    - Recommended action
    - Assessor decision
- Highlight differences between the two cases in amber

BENCHMARK PANEL (below the comparison):
- Call the API one more time with both cases as context and ask:
  "You are comparing two insurance claims that have both been
   through FWA review. Identify the 3 most significant differences
   in their risk profiles, explain what they suggest about each
   case, and identify if either case might benefit from being
   reviewed in light of the other."
- Display this as a short AI-generated analysis paragraph with
  the title "AI Comparative Analysis"
```

## If Something Goes Wrong

Things will break. An agent might return an error. The results panel might be blank. This is completely normal. Here is how to handle it:

### Step 1: Describe the problem

```text
Something is wrong. The agents ran but the results page is blank —
no cards are showing. The loading indicator completed successfully.
Can you check the code that parses the API response and displays
the data?
```

### Step 2: Let Claude Code diagnose and fix

Claude Code will look at the code, identify the issue, and fix it. You do not need to understand what went wrong technically.

### Step 3: If it is still broken, give more context

```text
It's still not working. Here is what I see: [describe exactly what
you see on screen]. Here is the error message from the browser:
[paste any error text]. This started happening after [describe
the last change made].
```

### Step 3.5: Clear the conversation if Claude is going in circles

If Claude seems to be repeating the same fix, type `/clear` to reset the conversation. This gives Claude a fresh perspective on the current state of your files without the baggage of a long debugging history. Your files on disk are not affected — only the chat history is cleared.

### Step 4: If you are stuck, start a fresh section

```text
Let's rebuild the results dashboard from scratch. Keep the upload
screen, the agent pipeline, and the investigationResults variable,
but rewrite everything that displays the results. Use a simpler
approach this time.
```

**The golden rule: Do not touch the code.** Describe what you see, what you expected, and when it started. Claude Code will fix it.

| Problem | What to do |
|---------|------------|
| Blank results / no data | Tell Claude Code: "The agents ran but the results page is blank. Can you check the code that generates the simulated responses?" |
| Layout looks broken | Tell Claude Code: "The layout looks wrong — [describe what you see]. Can you fix it?" |
| Claude Code going in circles | Type `/clear` in Claude Code to reset the chat. Your files are safe. |
| Agent returns error | Tell Claude Code: "I'm getting this error: [paste error]. Fix it." |
